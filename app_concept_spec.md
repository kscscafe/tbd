## Concept

### One-liner
**断片を投げ込むだけで、AIが整理・分類・エージェント化し、論理的な成果物として仕上げてくれる。**

### Origin Story
2026年4月、杉崎は Claude のAIエージェントを7体作り、自分の仕事をそれぞれに任せ始めた。結果、これまで中途半端で終わっていた仕事が次々と完成品として仕上がるようになった。

頭の中には常に断片が絡み合って存在していて、これを少しずつしか出力できなかった。AIが断片を整理し、論理的につなげてくれることで、初めて方向性と形が立ち上がった。これにより：

- 頭の中で覚えておかなくてよくなった
- 管理を AI に任せて頭が軽くなった
- 中途半端だったものが完成品になった

**この体験を、エージェントの設計・管理を意識せずに誰でも再現できるようにする — それが TBD。**

### Background
人間の頭の中には豊かで複雑な思考がある。しかし ASD・ADHD をはじめ、多くの人が「整理・分類・構造化」を苦手とする。既存ツール（Notion、Obsidian、フォルダ管理）はすべて「自分で分類してください」前提で、それが能力発揮を妨げる。

Claude Projects・Gemini Gems は強力だが、自分でエージェントを設計・運用する必要があり、その時点で離脱する人が多い。

**TBD は、設計を意識させずに「Claude Projects 級の体験」を提供する。**

### Core Metaphor
> 「考えは溢れてる。整理だけが邪魔をしてる。」
> 「7人のAIエージェントを、設計せずに勝手に立ち上がるようにする。」

---

## The Problem

### 既存ツールの限界
- Notion / Obsidian → 「自分で構造を作れ」前提
- Claude Projects / Gemini Gems → 「自分でエージェントを設計しろ」前提
- マインドマップ系 → 静的・進捗や状態を表現できない
- タスク管理（Linear / Asana） → 断片入力 → 自動分類のフローがない

### 個人ユーザーの本当の痛み（ASD 視点）
1. **思考を見失う**：「先週何を考えていたっけ」
2. **完結できない**：いろんな話が中途半端に並列している
3. **自分の傾向が見えない**：何に時間を取られているかわからない
4. **全体像が見えない**：各分野の進捗・関連性・優先度が一望できない
5. **整理という行為自体に消耗する**

---

## The Solution

### フロー
```
1. 断片を投げ込む（テキスト・音声・ファイル）
      ↓
2. AIが自動でツリーに振り分ける（auto-grow）
   - 既存ノードに追加
   - 既存ノード配下に新ノード作成
   - まったく新しい最上位ノードを作成
      ↓
3. 各ノードがエージェントとして育つ
   - そのテーマの system_prompt
   - 蓄積された文脈
   - 進捗 / 状態 / 優先度 / next_action
      ↓
4. 全体俯瞰ビューで一望
   - ツリー構造で親子孫が見える
   - 進捗・関連性・優先度が可視化
      ↓
5. 各エージェントに仕事を任せて成果物を出す
   - 普段はチャット内で生成
   - 必要なら artifacts に昇格して保存・履歴管理
```

### 2層インターフェース

**入力面（軽い・低摩擦）**
- 断片を投げ込むだけ
- フローティングボタン or 音声入力
- 分類は AI に丸投げ

**俯瞰面（重い・構造的・常時可視）**
- 全エージェントが Mission Control 的に一望できる
- ノードごとに：進捗 / 状態 / 優先度 / next_action / 関連性
- ドリルダウンしても全体が見失われない

### キー思想
1. ユーザーは投げ込むだけ。分類は AI の仕事。
2. ツリー構造で全体俯瞰常時可視。
3. AI が auto-grow、ユーザーは手動 D&D で再配置可能。
4. 各ノードは「エージェント」として育ち、文脈を蓄積する。
5. 成果物は会話の中で生成、保存したいものだけ artifacts に昇格。

## Data Model（概念）

```
nodes（自己参照ツリー）
  ├─ id, parent_id
  ├─ name, system_prompt
  ├─ status（active / waiting / blocked / dormant / done）
  ├─ progress（0-100）
  ├─ priority（user_set or ai_estimated）
  ├─ next_action（AI が提案する次の一手）
  ├─ auto_named（AI 命名 or 手動命名）
  └─ created_at, updated_at

fragments（生の入力）
  ├─ id, user_id
  ├─ raw_text, source（text / voice / file）
  ├─ node_id（自動配送先、手動移動可）
  └─ created_at

outputs（成果物アーカイブ）
  ├─ id, node_id
  ├─ type（提案書 / メール / 議事録 / 自由）
  ├─ title, content
  ├─ source_fragment_ids
  ├─ version
  └─ created_at

users
  └─ 認証は otera-chat 流の HMAC セッショントークン
```

## Core Features (MVP)

### 1. 断片インボックス
- テキスト・音声で投げ込む（音声は将来）
- 分類・タグ付け不要
- フローティングボタンで瞬間入力

### 2. AI ツリー自動成長
- 投げ込まれた断片を AI が読み解く
- 既存ノードへの追加 / 子ノード作成 / 新ルートノード作成 を判断
- 階層を考慮した cluster.js 拡張版

### 3. ノード = エージェント
- 各ノードに system_prompt
- そのノード（およびサブツリー）の文脈で対話
- 親ノードを呼ぶと子孫の文脈も参照される

### 4. 全体俯瞰ビュー
- 3 ペイン構成：
  - 左：インデントツリー（ナビ）
  - 中央：簡易ダッシュボード（進捗・状態の一覧、将来サンバースト/ツリーマップ）
  - 右：カレントノードの詳細・断片リスト・チャット
- 各ノードに進捗 / 状態 / 優先度 / next_action を表示
- 関連性は線で描画（pgvector の埋め込みベース、Phase 2 以降）

### 5. ノード操作
- 手動 D&D で別の親に移動
- 名前変更（→ auto_named=false）
- マージ・分割
- ステータス変更

### 6. 成果物生成（C ハイブリッド）
- 通常はチャット内で生成
- 「artifact として保存」ボタンで `outputs` テーブルへ
- ノード詳細画面に「📦 成果物」タブ

## Tech Stack

- **Backend:** Vercel Functions（Node.js / ESM、otera-chat 流用）
- **DB:** Neon PostgreSQL + pgvector
- **AI:** Anthropic API（claude-sonnet-4-6）
- **Frontend (MVP):** 素 HTML/CSS/JS（otera-chat の構成を流用・改造）
- **音声:** Whisper API or iOS Speech Framework（将来）
- **モバイル:** Flutter or Swift（バックエンド完成後に判断）
- **認証:** HMAC 署名セッショントークン（otera-chat の `_lib.js` を流用）
